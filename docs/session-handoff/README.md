# AI Session Handoff

**ctx 耗盡或使用者指示時，按此流程交接。**

---

## 資料夾結構

| 檔 | 讀者 | 用途 |
|---|---|---|
| `main-scene-playbook.md` | **原主場景** | 交接全流程 SOP + 各階段範本（主場景消化 + 客製化產指派）|
| `business-derivation.md` | **新場景 + 使用者 + 原主場景** | 業務骨架（廳主視角推導 + 系統設計哲學 + 協作模式概述）|
| `README.md` | 使用者 / 管理者 | 本檔 |

## 核心設計

### 分片保護

範本在**原主場景手裡**（`main-scene-playbook.md`）。新場景**永遠不開範本**，只看當下指派的 prompt + 必讀文件（如 CLAUDE.md / business-derivation.md / DECISIONS.md / collab-framework/*）。

### 探索升級三階段（新場景做）

```
自己推演 → graphify 掃疑點 → 使用者補業務缺口
```

每階段用**更精細工具**收斂盲點：文件 → 圖譜 → 業務方。

### 主場景 ctx 曲線

```
階段 0（重度消化：首訊息 + 起始 prompt，核心產出）
  ↓
階段 1（等回報）
  ↓
階段 2（中度：寫題目）
  ↓
階段 3-6（輕度：短指派 / 一句話）
  ↓
階段 7（脫手）
```

## 為什麼這樣設計

傳統思路是「交接文件越詳盡越好」。但 LLM 不是人 — 有現成答案就複製、看到完整流程就按劇本走、傾向編織流暢敘事掩蓋不確定、資訊過量會偷懶。

所以每個設計都在**反向利用**這些特性，逼出真正的內化：

| 設計 | 對應的 LLM 特性 | 應對 |
|---|---|---|
| 分片保護（範本只給原主場景） | 看到完整流程就按步驟走，失去當下判斷 | 剝奪上帝視角 — 每階段對新場景都是當下思考，不是演劇本 |
| 交接指令用 message 而非 summary | Summary 的「全貌」容器誘發主場景寫結構化後續步驟 — 尤其 ctx 耗盡時判斷力下降更嚴重；且多燒一份 ctx | 首訊息用 message 發出，「下指令」語氣自然聚焦當下，不易滑向寫全景 |
| 首訊息只指向原文，不濃縮內容 | 有現成答案就複製，不讀原文 | 首訊息只列必讀清單 + 回報指示（`"you have nothing, go read"`） |
| 範本 ≠ 指派，強制消化 + 客製化 | 轉貼 = 沒處理本輪情境 | 原主場景用 ctx 把範本與脈絡融合 — AI 的價值在消化 |
| 探索升級三階段（推演 → graphify → 使用者） | — | 成本遞增收斂：推演零成本 → graphify 技術事實 → 使用者唯一業務來源。最貴的人力只用在 dead-end |
| 主場景 ctx 曲線（0 重、1-7 遞減） | 原主場景 ctx 快滿，不能再承擔重運算 | 核心產出壓縮在階段 0 一次爆發，後面都是等回報 / 一句話指派 — 平穩退場 |
| Close 舊交接 topic 開新 | context priming — 看到舊失敗紀錄會消沉被定錨 | 乾淨桌面才能全力思考 |
| 誠實標疑點 > 寫得漂亮 | LLM 傾向編織流暢敘事掩蓋不確定 | 把「標 dead-end」明確定義為**核心價值**，扭轉討好傾向 |
| 自檢三題（生命週期圖 + 哲學圖 + 場景推演） | 答摘要 ≠ 真讀 | 能重組畫圖 + 推演未見場景才算內化；Q3 還兼做工具邊界檢測 |
| 維護回補 business-derivation.md | — | 自演化系統，不是凍結 SOP — 每次暴露的業務缺口都固化進骨架 |

**本質**：刻意製造資訊落差的編排。原主場景擁有全景但即將退場、新場景有空缺必須自填、使用者只補最貴的業務缺口。三方資訊來源（範本 / 文件 / 業務方）成本與精度遞增。

把 LLM 交接當**工程可靠性問題**處理 — 不依賴「模型夠聰明會自己摸清」，而是用流程強制走過 **推演 → 自救 → 對齊** 三道防線。

---

## 紀律

- **範本 ≠ 指派**：原主場景消化 + 客製化，不轉貼
- **分片保護**：不劇透後續步驟（不列編號 / 不提工具名）
- **誠實標疑點** > 寫得漂亮
- **code / graphify 是 ground truth**
- **暫存檔放系統 `/tmp/`**（見 workspace CLAUDE.md）

---

## 維護（正向循環）

每次對齊發現盲點 → 回補 `business-derivation.md` / 對應文件 → 下次接手從更完整骨架出發。
