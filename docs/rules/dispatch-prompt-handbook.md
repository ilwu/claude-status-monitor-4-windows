# 發包 Prompt 寫作手冊 — 主 session 實作指南

> 主 session 在協作模式下做「發包」動作時的 prompt 寫作手冊。
> **協作契約**見姊妹 rule `multi-session-dispatch.md`（角色 / topic type / 回報格式 / 品質檢查時機 / Race 規則）。
> **專案特定**（guide 對照表 / 本地慣例 / 實例案例 / 專案執行約束）在各專案自有的 `dev-session-handoff.md`（本文不重述）。

---

## 一、核心理念

發包 prompt **不是任務說明書**，是**心智交接工具**。

接發包的 session 沒讀過對話歷史、沒踩過你踩過的坑、不知道哪些慣例約定俗成。Prompt 的品質決定這個 session 能不能接上你的 context 繼續推進，而不是從零探索。

三個基本信念：

1. **給文件、不給結論** — 必讀的設計文件比你的摘要權威
2. **列經驗、不列命令** — 踩過的坑值得被記下，讓 session 知道「為什麼不能這樣」
3. **給範圍、不給自由** — 明確劃界能防止 session 擴大範圍變成災難

## 二、7 大類原則

### 2.1 必讀文件分層

把必讀文件**按「讀的順序」分層**，不是條列一堆讓 session 自己挑：

```
1. 任務主文件（設計文件 .md）      ← 核心
2. 任務類型對應 guide             ← 專業領域規範（見各專案對照表）
3. 架構哲學 / 基礎設施指南          ← 為什麼這樣設計
4. 近期正確範本（同類已完成任務）   ← 怎麼做才對
5. 專案 CLAUDE.md + CHECKLIST     ← 硬性規則
```

**為什麼要給「正確範本」**：設計文件描述「應該怎樣」，但 session 落地時仍會有格式困惑。給近期剛完成的同類任務實作，session 看到實際碼，format 跟著學，比文字說明有效。

**任務類型對應 guide 對照表**：各專案自訂，見該專案 `dev-session-handoff.md`。

### 2.2 業務理解摘要（5-10 行）

設計文件可能 300-500 行，但真正需要 session 對齊的業務規則只有幾條。摘要放在 prompt 裡：

**好例子**：
```
- 2621 = 流水帳呈現器，唯讀（不做寫入、不做線下結清）
- 資料來源：單一 table，禁止 join 其他 table 拼湊
- 即時餘額從 list[0].after 取，不設獨立 API
- 事件類型用業主視角翻譯
- 金額單位：元（自動轉換機制 ÷100，Service 不手動轉）
```

**壞例子**：
```
請參考設計文件做代理佣金明細頁面。
```

### 2.3 關鍵實作點前置提醒

找出任務中「新手容易做錯」的 2-5 個點，明確寫出「要用 X，不要用 Y」。

#### 2.3.1 設計文件「沿資料流清單」驗過（主 session 自查）

寫設計文件時，**對照實際資料流走一遍**，每個經手的 DTO / Service / 寫入點都要列清楚。

**經驗**：設計文件 §六 漏列中間 DTO / trigger base — 實際寫入要從 SubModule → CalculateResult → trigger.processSource → LedgerEntry 才能傳到 ledger。只列最末端的 Entity 改動不夠，session 要靠自探補完才行。

**檢核方法**：從資料源一路追到寫入點，畫出完整路徑，每個檔案都要列入發包。

### 2.4 踩坑預告

把過去發現、未來可能重踩的坑**明確列出**，加上應對方式。

救一個 session 平均一小時（不提 → session 自己踩 → 回報問題 → 你澄清 → session 回頭改）。

#### 2.4.1 主 session 寫 prompt 前先查系統慣例

不查慣例 → 設計文件寫死某方向 → session 按設計做 → 使用者發現不對 → 返工。

**對策清單**（寫設計文件前查）：
- [ ] 同類頁面的匯出格式 / 使用哪個函式庫
- [ ] 日期時間格式慣例（本地時區 / UTC / 格式化函式）
- [ ] UI 元件選擇（既有共用元件）
- [ ] i18n key 命名慣例
- [ ] 金額顯示格式（自動轉換機制 / 負值顏色 / 幣別符號）
- [ ] 權限常數拆分慣例
- [ ] 錯誤訊息結構（Exception builder + i18n detail）

規則：**如果「我不確定」→ 去看同類頁面**。不要憑感覺寫設計，讓 session 實作後才被糾正。

#### 2.4.2 上線前舊資料直接清，不為相容妥協新邏輯

**原則**：系統尚未正式上線時，測試區舊資料是**拋棄性的**，不為它們相容而犧牲新架構清晰度。

**判斷標準**：
- 已上線有真實使用者 → 嚴格相容 + migration
- **未上線（測試區、本機）→ 直接清**
- 混合狀態 → 和使用者對齊哪些清哪些留

#### 2.4.3 批次修改失敗時不要嘗試 `clean 重來`

**絕對不要**在改一半出錯時跑 `git checkout .` / `git reset --hard` / `git clean -fdx` 「回到乾淨狀態重來」。這會清掉 working tree 所有改動，包括 `.env` 等 untracked 檔，**無法救回**。

**正確做法**：
- 逐個修，不放棄累積的進度
- 真的要放棄改動 → **停下來回報使用者**，由使用者決定 stash / commit / 丟棄
- 批次操作前先 `git status` 確認 working tree 乾淨

**防護層**：各專案應配 PreToolUse hook 硬擋破壞性 git。

### 2.5 範圍邊界 + 越界拒絕授權

明確寫「不改範圍外的東西」，並授權 session 拒絕越界。

**例子**：
```
- 不改範圍外的東西：Ledger 模組內部邏輯、其他 admin 頁面、錢包模組內部全部不動
- 有疑問先問，不自作主張擴大範圍
```

**為什麼要明確授權**：好的 session 會主動質疑 prompt 的邊角，若沒授權會糾結要不要動。明確授權後 session 敢提問題。

### 2.6 Phase 拆分 + 回報節奏

子任務切成 5-9 個 Phase，每 Phase 一個驗證點：

```
Phase 1：Entity + DTO 改動 → tsc 0
Phase 2：Service 實作 → tsc 0
Phase 3：Controller + Module → tsc 0
Phase 3.5：★ 主程序品質檢查（見下方）
Phase 4：寫基礎場景測試 → 綠
Phase 5：批次寫剩餘場景 → 全綠
Phase 6：前端 → build 過
Phase 6.5：★ 前端品質檢查（可選）
Phase 7：i18n
Phase 7.5：使用者瀏覽器驗證
Phase 8：code-reviewer 第二輪
Phase 9：commit
```

**禁衝底**：每個 Phase 回報一次，不要一路衝到底。出錯時 rollback 範圍小、可即時調整、進度精確掌握。

#### 2.6.1 品質檢查時機 + 回報格式

**完整規則見 `multi-session-dispatch.md`**：
- §七之二 品質檢查兩輪時機（第一輪主程序完成、第二輪 commit 前）
- §五 回報格式（`mmsg topic-add` + 對自己 Claude 介面輸出皆標 topic id）

本文不重述，避免維護兩份衝突。

#### 2.6.2 修主邏輯後回檢測試假設

修 bug 後**回檢所有測試的隱性假設**。

**經驗**：code-reviewer 找到 bug 修完後，測試 fixture 中「base 時間用 Date.now() + i*1000 產生未來時間」被新邏輯的時間判斷擋掉 → 測試掛掉。

**防範**：
- 修過濾、排序、時間邏輯後，回跑**所有**相關測試
- 看測試 fixture 對「當前時間、時區、排序」等隱性依賴
- 測試掛掉不代表測試錯，**先檢查是不是 fixture 假設跟著 bug 改變了**

### 2.7 執行約束（通用部分）

以下每份發包 prompt 都必須有：

- 不趕進度
- tsc 零錯誤才進下一步
- 測試必寫 + 必跑
- 不啟動 dev server（使用者管理）
- commit 預設略過文件（`.md` / `docs/**`）— 具體規則依各專案 CLAUDE.md
- 絕不自行做破壞性 git 操作（搭配專案 PreToolUse hook 硬擋）
- 品質檢查發現**小而確定**的問題 → 直接修，不推給下包
  - 範圍：typo / 未用 import / 明顯 bug 1-2 行 / 格式 / 補缺失型別
  - 例外（仍回報不自己修）：大型 refactor / 改 API 簽名 / 新業務規則 / 多檔連動 / 修法不唯一 / 不確定業務意圖
  - 原則：最終責任在使用者身上，session 之間的職責切分不是拖延理由

**專案特定執行約束**（如特定 git command / 測試指令 / kill-jest 等）：各專案 `dev-session-handoff.md` 補充。

## 三、Prompt 骨架模板

### 3.1 完整模板（可直接複製）

```
你是{{專案名稱}}的{{session 角色}}，接手{{任務描述}}。

## 任務

{{1-2 句任務描述}}

## 必讀文件（按順序）

1. `{{設計文件路徑}}`
   {{這份文件在任務中扮演的角色}}

2. 任務類型對應 guide 清單（見專案 `dev-session-handoff.md` 對照表）

3. `{{架構哲學 / 基礎設施指南路徑}}`
   {{為什麼要讀這份}}

4. `{{近期正確範本路徑}}`
   {{同類任務實際案例}}

5. `CLAUDE.md`（專案根目錄）
   critical-rules 特別重要。

6. `{{編碼規範路徑}}` + `CHECKLIST.md`
   每次修改對照。

## 執行約束

見 `multi-session-dispatch.md` §七之二 品質檢查 + 本專案 `dev-session-handoff.md` 執行約束段。

## 關鍵業務理解

{{濃縮 5-10 行業務規則}}

## 關鍵實作點

{{3-5 條最容易誤實作的地方 — 明確「用 X 不用 Y」}}

## 特別提醒（踩坑預告）

{{過往踩過、未來可能重踩的 2-5 條}}

## 測試位置

{{e2e 目錄路徑 + TestModule 位置 + 組織原則引用}}

## 回報要求

見 `multi-session-dispatch.md` §五 回報格式：
- `mmsg topic-add` 第一行 `# [topic-id] P? 狀態 — 摘要`
- 對自己 Claude 介面輸出階段回報也標 topic id

每個 Phase 回報一次，不要一路衝到底。

## 開始前

1. 讀完以上文件 → 重述理解（5-8 句話）→ 列第一步 → 徵得確認再動工
2. 動工前跑 git status 確認 clean
3. 主動指出 prompt 沒涵蓋的邊角，不確定時問清楚再動

不要跳過理解階段直接寫碼。
```

### 3.2 規模差異的調整

**小任務（~0.5 天）**：必讀 4 份 / Phase 5-7 個 / 踩坑預告 1-2 條

**中大任務（2-5 天）**：必讀 5-7 份 / Phase 8-10 個 / 踩坑預告 3-5 條 / 範圍邊界列更細

## 四、收尾流程（主 session 執行）

發包 session 完成任務後，主 session 還需要：

### 4.1 i18n 收尾（若任務涉前端）

- 發包 session 只做核心語系（如 zh-CN / tw / en）
- 主 session 用 i18n agent 批次翻譯其他語系
- 具體 agent 或工具 依專案

### 4.2 sys_menu 同步（若新增 admin 頁面）

- 本機 DB + 測試區 DB 都要同步
- 具體 mongosh script / 自動化腳本依專案

### 4.3 DB Migration（若涉 entity 新欄位）

- 發包 session 提供 script
- 主 session 負責執行本機 + 測試區

### 4.4 Reviewer × 2

- 時機見 `multi-session-dispatch.md §七之二`
- 具體帶什麼文件 / 怎麼跑依專案

## 五、發包 session 的主動權（授權清單）

讓接發包的 session 知道他**可以且應該**主動做的事：

1. **主動指出 prompt 沒涵蓋的邊角**（sys_menu / role 拆分 / migration）— 補漏比盲目執行有價值
2. **質疑設計文件的矛盾**（實際落地時才會發現的衝突）— 停下來問
3. **調整 Phase 拆分**（實作發現某 Phase 太大可拆）— 回報並對齊
4. **動工前跑 git status 兩邊 repo**（避免誤動別 session 的 WIP）
5. **發現既有架構 bug 時先停下**（不自行擴大範圍修，列為 TODO 回報）

這不是給 session 自由決定，是**給 session 心智空間**。發現問題立即停下來對齊，比悶著頭衝完再返工有效。

## 六、持續增補

每次發包完成後，把**新發現的坑 / 好用的模式**回寫：
- 跨專案通用 → 寫回本文件
- 專案特定 → 寫回該專案的 `dev-session-handoff.md`

容忍本文件變長，因為每一條都換一個小時的返工。

## 附錄 A：發包 Prompt 檢核清單

### A.0 任務含測試時的強制必讀

凡發包含 e2e 測試 / 整合測試 Phase，必讀清單**必須**包含該專案的**測試場景規範文件**（路徑依各專案而定）。

測試場景規範通常定義：
- 測試目錄結構（domain / feature 兩層）
- 檔案命名慣例
- Controller / spec 雙軌架構（若採用）
- TestData / Builder / Result 模板
- 場景實作紀律（一次一場景、獨立清理、不依賴順序）

沒列 → session 靠既有程式碼模仿 → 目錄結構 / 檔名慣例 / 斷言方式可能偏離規範 → 需重建（損失和測試已寫的多寡成正比）。

**檢核原則**：發包若含任何「測試 Phase」，必讀清單第一順位放該專案的測試場景規範。各專案 `dev-session-handoff.md` 應在其對照表標出此文件位置。

### A.1 主 session 寫設計文件前

- [ ] 查過系統慣例（匯出格式 / 日期格式 / UI 元件 / i18n key 命名 / 權限拆分）
- [ ] 設計文件「沿資料流清單」驗過（從源頭追到持久化，每個 DTO/Service 都列）
- [ ] 近期同類完成頁面可當範本

### A.2 主 session 寫 prompt 前

- [ ] 必讀文件按順序排
- [ ] 有列近期正確範本
- [ ] 業務理解摘要 5-10 行
- [ ] 關鍵實作點 3-5 條
- [ ] 踩坑預告 2-5 條
- [ ] 範圍邊界明確
- [ ] Phase 拆分 5-9 個 + 每 Phase 驗證點
- [ ] 執行約束（tsc / 測試 / 不啟 server / commit 規則 / 破壞性 git hook）
- [ ] 測試位置寫清楚
- [ ] i18n 策略（若涉前端）
- [ ] 回報節奏（每 Phase 回報 + 完成回報格式 + 標 topic id）
- [ ] 動工前檢查（git status + 重述理解）
- [ ] 授權主動補邊角

### A.3 發包回報後（主 session 收尾）

- [ ] i18n 補齊（若涉）
- [ ] sys_menu 測試區同步（若涉）
- [ ] 主線可追問項目列 TODO
